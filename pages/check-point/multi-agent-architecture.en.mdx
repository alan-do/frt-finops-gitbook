# Multi-Agent Architecture powered by FOCUS

<Callout type="info" emoji="🤖">
  **Autonomous FinOps Vision:** The FRT FinOps system eschews rigid data processing scripts in favor of a **Multi-Agent** architecture operating seamlessly on a foundation of data normalized by the **FOCUS Open Standard**. This combination creates a "Virtual Team" capable of communicating in a universal data language, automating the entire workflow from ingestion and anomaly detection to root-cause analysis.
</Callout>

The Multi-Agent architecture enables the system to operate autonomously, where each Agent assumes a specific role and makes decisions based on real-time data extracted from the FOCUS Data Lake.

## 2. The Role of Each Automation Agent in the FOCUS Ecosystem

The system comprises 4 primary Agents, guided by the FOCUS standard for interaction:

<Cards>
  <Card icon="🕵️" title="Data Ingestion Agent" href="#">
    **Task:** Universal Data Translator.  
    Continuously calls APIs to AWS, Azure, and OpenAI to pull usage logs. It automatically **transforms** vendor-specific data columns into the precise format of the FOCUS column library (e.g., aggregating into `BilledCost` and `ProviderName`).
  </Card>
  <Card icon="🚨" title="Anomaly Detection Agent" href="#">
    **Task:** Budget Gatekeeper.  
    Because the data is already normalized, this Agent only needs to run a single Machine Learning model on the `BilledCost` column to monitor the entire organization in real-time. It triggers an alarm if deviations from the baseline are detected.
  </Card>
  <Card icon="💡" title="Optimization Analyst Agent" href="#">
    **Task:** Cost Optimization Advisor.  
    Scans historical FOCUS data to find idle resources based on standard tags (`ChargeCategory`, `ResourceName`) and proposes saving strategies (e.g., "Purchase a Savings Plan for this resource bucket").
  </Card>
  <Card icon="🗣️" title="Reporting & Comm Agent" href="#">
    **Task:** GenAI Spokesperson.  
    Configured with Prompts designed to read and understand FOCUS data tables. Receives signals from other Agents, explains the reasons for cost increases, and writes friendly reports sent to Internal Chat Channels.
  </Card>
</Cards>

## 3. Practical Workflow: Incident Handling on FOCUS Data

To better visualize the power of combining Multi-Agent and FOCUS, below is the workflow when a cost spike incident occurs:

<Steps>
### Step 1: Continuous Ingestion & Normalization
The **Data Ingestion Agent** detects a spike in API Usage by Product Team X for OpenAI services. It pulls the data, immediately transforms it into standard FOCUS rows, and pushes it into the Data Lake.

### Step 2: Real-time Detection & Alerting
The **Anomaly Detection Agent** monitoring the Data Lake notices the `BilledCost` column for Team X's `SubAccountId` has increased 300% compared to the 7-day average. It instantly creates an `Alert Event` and sends it to the queue.

### Step 3: Deep Root-Cause Analysis
The **Optimization Analyst Agent** receives the `Alert Event`. It writes a standard FOCUS SQL query to dive deep into Team X's logs and discovers the cause originates from the `ServiceCategory` = "AI API". It packages this root cause.

### Step 4: Actionable Human Alert
The **Reporting Agent** receives the "Root Cause" from the Analyst. It uses an LLM to compose an urgent, user-friendly message:
> *"🚨 **Alert:** Product Team X has exceeded costs by 300% (Spent $500 in the last 2 hours based on standard BilledCost). Suspected cause: AI API services are being called continuously. Please check your systems immediately!"*
(Sent via internal communication channel)

### Step 5: Auto-Remediation (Optional)
If granted Actionability permissions, the Orchestrator can automatically call an API to temporarily rate-limit Team X's account until developers fix the bug, preventing further waste.
</Steps>

## 4. Organizational Value of this Integration
Building a Multi-Agent system on a FOCUS foundation helps FRT:
1. **Universal AI Models (Write once, run anywhere):** Agents (for anomaly detection and analysis) only need to be trained/prompted once on the FOCUS format, and they automatically work correctly across all providers (AWS, Azure, SaaS) without rewriting code.
2. **Eliminate Bottlenecks:** The FinOps team is not overwhelmed by translating the billing terminology of each individual vendor.
3. **Flexible Scalability:** When a new Cloud Vendor is added, only the data mapping rules for the `Data Ingestion Agent` need updating; the rest of the Multi-Agent system continues to function normally.

## 5. Implementation Challenges & Risks (MVP Considerations)
While highly feasible, implementing this architecture with **1.5 FTE** resources presents the following practical risks:
*   **Data Latency:** Cloud Billing data is typically delayed by 12-24 hours. Therefore, "Real-time Detection" is practically "Near Real-time" (1-day delay). True real-time alerting requires integration with system logs (which is resource-intensive).
*   **AI Hallucination & Auto-Remediation Risks:** LLMs can infer incorrectly. Granting AI the authority to automatically shut down servers (Actionability) is extremely dangerous. For the MVP, a **Human-in-the-loop** model (AI proposes, human approves) is mandatory.
*   **Cost of AI (Cost of FinOps):** Continuously feeding massive log files into GPT-4 for analysis will result in exorbitant API costs. Solution: Only use AI (LLMs) for analysis *after* cheap statistical algorithms (like Z-Score) have detected an anomaly.
*   **State Management:** Robust Retry/Error Handling mechanisms between Agents are required to prevent a domino effect (e.g., a network error in the Ingestion Agent causing the Reporting Agent to output false information).
